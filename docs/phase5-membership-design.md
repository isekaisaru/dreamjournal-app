# YumeTree Phase 5-0 メンバー機能 MVP設計

作成日: 2026-05-27
ステータス: 設計ドキュメントのみ

## Phase 5の目的

Phase 5では、YumeTree を「ひとりで安心して使えるAI夢ノート」でありながら、必要なときだけ大切な相手と一部の夢を共有できる体験に拡張する。

夢の本文はプライベート性が高いため、実装より先に安全なMVP範囲を定義する。最優先するのは、誤公開を防ぎ、承認された相手だけが明示的に共有された夢を閲覧できること。

このドキュメントは Phase 5-0 の設計整理であり、migration、DB変更、Rails API実装、フロントエンド実装は行わない。

## ユーザーストーリー

- ユーザーは、夢を共有したい相手を1人招待できる。
- 招待された相手は、招待を承認または拒否できる。
- 招待が承認された場合のみ、ユーザーは一部の夢を `partner` として共有できる。
- `private` の夢は、招待状態に関係なく本人だけが閲覧できる。
- `pending`、`rejected`、`removed` の相手には、夢本文を見せない。
- ユーザーは夢詳細画面で、その夢が自分だけに見えるのか、承認済みの相手にも見えるのかを確認できる。

## MVPでやること

- 1人の相手を招待できる関係性を設計する。
- 招待状態として `pending`、`accepted`、`rejected`、`removed` を扱う。
- `accepted` 後に関係解除できるようにし、解除後は相手からの閲覧権限を即時失効させる。
- 夢の公開範囲は `private` と `partner` の2種類に絞る。
- `dreams.visibility` の default は必ず `private` にする。
- `partner` の夢は、`accepted` な相手だけが閲覧できる。
- ID直打ちでも、Rails API側で必ず閲覧権限を確認する。
- UI上で共有状態を明示し、誤公開を防ぐ。

## MVPでやらないこと

- `public` 公開
- コメント機能
- 通知
- メール送信
- 外部共有URL
- 複数メンバー、家族グループ、共有先の個別選択
- Stripe連携
- Phase 4「夢画像シェアカード」の修正
- Dependabot対応

## データモデル案

### memberships

Phase 5実装時に追加を検討するテーブル。

| カラム | 内容 |
|---|---|
| `id` | 主キー |
| `inviter_id` | 招待したユーザーの `users.id` |
| `invitee_id` | 招待されたユーザーの `users.id` |
| `status` | `pending` / `accepted` / `rejected` / `removed` |
| `created_at` | 作成日時 |
| `updated_at` | 更新日時 |

`inviter_id` / `invitee_id` は招待の向きを表す。ただし関係性としては、AがBを招待した場合と、BがAを招待した場合を同じ2人の関係として扱う。

承認後の関係は双方向に扱うため、閲覧権限チェックでは「自分が inviter か invitee か」に依存しすぎないようにする。DB側では `UNIQUE(LEAST(inviter_id, invitee_id), GREATEST(inviter_id, invitee_id))` のような一意制約を検討し、Rails側でも同等のバリデーションを追加する。

### dreams

Phase 5実装時に追加を検討するカラム。

| カラム | 内容 |
|---|---|
| `visibility` | `private` / `partner` |

`visibility` の default は `private` とする。既存の夢に対しても、移行時は明示的に `private` として扱う。

## `partner` の意味

MVPにおける `partner` は、1人の恋人や配偶者だけを意味する言葉ではなく、「招待を承認済みの相手」を指す。

- `accepted` な相手: `partner` の夢を閲覧できる。
- `pending` の相手: 閲覧できない。
- `rejected` の相手: 閲覧できない。
- `removed` の相手: 閲覧できない。

`removed` になった時点で、過去に `partner` として共有されていた夢へのアクセスも即時に失効する。

将来、複数メンバーや家族グループに拡張する場合は、`partner` という表示名・内部名が狭すぎないか再検討する。

## 権限ルール

- 自分の夢は、自分が常に閲覧できる。
- `visibility = private` の夢は、本人のみ閲覧できる。
- `visibility = partner` の夢は、本人と `accepted` な相手のみ閲覧できる。
- `pending`、`rejected`、`removed` の相手には、夢本文を絶対に返さない。
- 夢詳細APIでは、ID直打ちでも必ず閲覧権限チェックを通す。
- 非権限者が夢詳細APIにアクセスした場合は `404 Not Found` を返す。
- `403 Forbidden` は夢の存在を推測させるため、非公開夢へのアクセス拒否には使わない。
- 夢一覧APIでは、自分の夢と共有された夢を混ぜる場合でも、取得条件に権限チェックを含める。
- 更新、削除、画像生成、シェアカード生成など、夢に関連する操作は所有者本人だけに限定する。
- 共有された相手は、MVPでは閲覧のみ可能とする。

## 一意制約・バリデーション案

- `inviter_id != invitee_id` とし、自分自身を招待できないようにする。
- A→B と B→A は同じ2人の関係として扱い、重複した招待を作れないようにする。
- DB側では `UNIQUE(LEAST(inviter_id, invitee_id), GREATEST(inviter_id, invitee_id))` のような一意制約を検討する。
- Rails側でも、`inviter_id` と `invitee_id` を正規化した組み合わせで同等の一意性バリデーションを追加する。
- `pending` と `accepted` が同じ2人の間で並存しないようにする。
- `rejected` や `removed` がある場合の再招待可否は、既存レコードの扱いを含めて実装前に決める。
- `status` は定義済みの値だけを許可する。
- `accepted` な相手がすでにいる場合、MVPでは別の相手を追加できないようにする。

## ステータス遷移案

MVPでは、状態遷移を小さく保つ。

```text
pending -> accepted
pending -> rejected
accepted -> removed
```

- `pending` は招待中。相手には夢を見せない。
- `accepted` は承認済み。`partner` の夢だけ閲覧できる。
- `rejected` は拒否済み。相手には夢を見せない。
- `removed` は関係解除済み。相手には夢を見せない。
- `accepted -> removed` はMVPに含める。承認後に関係解除できない状態はセキュリティリスクになるため、解除導線と権限失効を最初から設計に含める。
- `removed` になった時点で、相手から `partner` の夢へのアクセスは即時失効する。

`rejected` 後の再招待を許可するか、拒否済みレコードを再利用するか、新しい招待を作るかはMVP後に検討する。

## API案

Phase 5以降で検討するAPI案。今回は実装しない。

| API | 目的 |
|---|---|
| 招待作成 | 相手を招待する |
| 招待一覧 | 自分が送った招待・受け取った招待を確認する |
| 招待承認 | `pending` を `accepted` にする |
| 招待拒否 | `pending` を `rejected` にする |
| 関係解除 | `accepted` を `removed` にする |
| 共有夢一覧 | accepted な相手から共有された `partner` の夢を取得する |

注意点:

- すべてのAPIでログイン必須にする。
- `accepted` 判定はフロントエンドではなくRails API側で行う。
- 共有夢一覧では、`visibility = partner` かつ `accepted` な関係だけを返す。
- `pending`、`rejected`、`removed` の関係では、夢の存在自体を推測しにくいレスポンスにする。
- 夢詳細APIで非権限者がID直打ちした場合は `404 Not Found` を返す。
- `403 Forbidden` は夢の存在を推測させるため、非公開夢の詳細APIでは使わない。

## UI案

- 設定画面に「相手を招待」導線を置く。
- 招待状態を `招待中`、`承認済み`、`拒否済み` のように明示する。
- 夢詳細画面に公開範囲を表示する。
- `自分だけ` / `承認済みの相手にも表示` のように、意味が分かるラベルにする。
- `partner` に切り替える操作では、誰に見えるかを相手名つきで表示する。
- `private` から `partner` に変更する際は、確認ダイアログを出す。
- 確認ダイアログでは `この夢は承認済みの相手にも表示されます` のように、誰に見えるかを明示する。
- 共有状態バッジを表示し、保存後も公開範囲を確認できるようにする。
- 初期値は常に `private` にする。

誤公開防止の文言例:

- `この夢は自分だけに表示されます`
- `この夢は承認済みの相手にも表示されます`
- `招待が承認されるまで、夢は相手に表示されません`
- `拒否済み・解除済みの相手には表示されません`

## セキュリティ・プライバシー注意点

- 夢本文はセンシティブ情報として扱う。
- default は必ず `private` にする。
- `public` はPhase 5 MVPでは扱わない。
- コメントはPhase 5 MVPでは扱わない。
- 外部共有URLはPhase 5 MVPでは扱わない。
- 共有された相手には、MVPでは閲覧だけを許可する。
- 夢の所有者以外に、編集、削除、画像生成、公開範囲変更を許可しない。
- APIレスポンスやエラー文言から、非公開の夢の存在が推測されにくいようにする。
- 非権限者への夢詳細APIレスポンスは `404 Not Found` とし、`403 Forbidden` は使わない。
- `removed` による関係解除後は、相手から `partner` の夢へのアクセスを即時に失効させる。
- フロントエンドの表示制御だけに頼らず、Rails API側で権限チェックを行う。

## 実装順序

| Phase | 内容 |
|---|---|
| Phase 5-0 | 設計ドキュメント |
| Phase 5-1 | `memberships` 基盤 |
| Phase 5-2 | `dreams.visibility` 追加 |
| Phase 5-3 | 共有夢一覧 |
| Phase 5-4 | UI改善 |
| Phase 5-5 | コメントなど拡張検討 |

### Phase 5-1: `memberships` 基盤

- migration、model、status enum、validationを追加する。
- A→B と B→A を同じ関係として扱う一意制約とバリデーションを追加する。
- 招待作成、一覧、承認、拒否、関係解除のAPIを最小実装する。
- この段階では夢の共有表示までは広げない。

### Phase 5-2: `dreams.visibility` 追加

- `dreams.visibility` を追加し、default を `private` にする。
- `private` / `partner` のみ許可する。
- 夢詳細APIで、`partner` の閲覧権限を追加する。

### Phase 5-3: 共有夢一覧

- accepted な相手から共有された夢だけを返す一覧を追加する。
- 自分の夢一覧と共有夢一覧を分けるか、UI上で混同しない表示にする。

### Phase 5-4: UI改善

- 設定画面で招待できるようにする。
- 夢詳細で公開範囲を確認・変更できるようにする。
- 共有状態バッジと誤公開防止文言を追加する。

### Phase 5-5: コメントなど拡張検討

- コメント、通知、外部共有URL、複数メンバー、家族グループを検討する。
- MVPの権限モデルが安定してから追加する。
