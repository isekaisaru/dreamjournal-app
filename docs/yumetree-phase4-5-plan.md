## Phase 4: 夢画像シェアカード

### 概要

AI生成画像を「シェア用カード」として表示・保存できる機能。夢の記録が「見せたくなるコンテンツ」になることで、口コミ誘導と登録率向上を狙う。

### ユーザーストーリー

夢を記録してAIが画像を生成した後、カード形式でまとめたビジュアルを1タップで保存・共有できる。

### UI設計案

```
┌─────────────────────────────────────┐
│  [AI生成画像]                        │
│                                     │
│  夢のタイトル                        │
│  記録日: 2026-05-18                 │
│                                     │
│  感情タグ: #不安 #空を飛ぶ          │
│                                     │
│  YumeTree  ユメツリー               │
└─────────────────────────────────────┘
     [画像として保存]  [リンクをコピー]
```

### 実装方針

- **フロントエンド**: `html2canvas` または `dom-to-image` でカードをPNG出力
- **バックエンド変更なし**: 既存の画像URL・夢データを使い回す
- **シェアURL**: 現時点では「ログイン必須で夢詳細ページへ」で十分。SNS直接共有は後回し
- **対象ページ**: `frontend/app/dreams/[id]/page.tsx` の画像表示部分に「シェアカードを作る」ボタンを追加

### 制約・注意事項

- AI生成画像は既存の `DallE3` / `gpt-image-1` フローを使う（新API追加なし）
- DB変更なし・migration不要
- html2canvas はSSRで動かないため `"use client"` 内のみで使う
- カード生成はクライアントサイドのみ（ファイルのサーバー保存は不要）

### 見積もり工数

- UIコンポーネント実装: 2〜3h
- html2canvas統合: 1h
- テスト・調整: 1h

## Phase 5: メンバー機能（ペア・家族向け）

### 概要

「ひとりでも、大切な人とも」という YumeTree のコンセプトを体現する機能。恋人・家族・友人と夢を共有し、相手の夢にコメントを残せる。

### ユーザーストーリー

- ユーザーAがBを「パートナー」として招待する
- BがAの夢記録を閲覧できる（Aが公開設定にした夢のみ）
- BがAの夢にスタンプまたは一言コメントを送れる

### データモデル設計（変更なし・メモのみ）

```
memberships テーブル（新規 — Phase 5実装時に追加）
  id
  inviter_id   → users.id
  invitee_id   → users.id
  status       enum: pending / accepted / rejected
  created_at / updated_at

dreams テーブルへの追加カラム（Phase 5実装時）
  visibility   enum: private(default) / partner / public
```

**現時点では DB 変更・migration は行わない。設計メモのみ。**

### UI設計案（フロントエンド）

1. **設定画面**: 「パートナーを招待」ボタン → メールアドレス入力 → 招待メール送信
2. **夢詳細画面**: 「シェア範囲」セレクター（自分だけ / パートナーにも見せる）
3. **ホーム画面**: 「パートナーの夢」タブ（パートナーが公開した夢リスト）
4. **コメント**: スタンプ絵文字 + 最大100文字の一言

### 実装順序（Phase 5着手時）

1. migration 作成（memberships, dreams.visibility）
2. Rails API: 招待・承認・夢公開範囲の CRUD
3. フロントエンド: 設定画面の招待フロー
4. フロントエンド: 夢詳細の公開範囲UI
5. フロントエンド: パートナーの夢タブ
6. フロントエンド: コメントUI

### 制約・注意事項

- **Phase 5実装時まで DB/migration は触らない**
- 招待メールは既存のメール送信基盤（ActionMailer）を使う
- プライバシー設計: デフォルトは `private`、パートナー公開はユーザーが明示的に選択
- Stripe課金との連動は Phase 5 では行わない（メンバー機能はフリープランでも使えるようにする想定）

### 見積もり工数

- DB設計・migration: 1h
- Rails API（招待・承認・閲覧権限）: 4〜6h
- フロントエンド全体: 6〜8h
- テスト: 2〜3h
- 合計: 約13〜18h
