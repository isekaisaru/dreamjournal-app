# 競合分析：ゆめのき vs yumelog.site

作成日：2026-05-16
更新日：2026-05-16

---

## 1. 発見の経緯と初期評価

2026年5月15日、「ユメログ」で検索したところ、同名のアプリ `yumelog.site` を発見。
同名・先発・機能豊富・低価格という点でショックを受けたが、
同時に「自分のアプリにしかない強みがある」ことも確認できた。

この発見を受けて **アプリ表示名を「ゆめのき」に変更する方針を決定**。
競合を研究対象として位置づけ、差別化軸を整理する。

## 3. 競合の強み（素直に評価する）

- **先発優位**：同名アプリとして認知を先に獲得している
- **価格の安さ**：ユーザー獲得ハードルが低い
- **機能の多さ**：ユーザーが期待する機能を幅広くカバー
- **UI/UX**：ユーザーが使いやすいと感じる点がある（要具体調査）

**学ぶべき点**：
- ユーザーが「夢アプリ」に何を求めているかのヒントがある
- UI設計・機能優先順位付けの参考にする
- 価格戦略（コストを下げながら継続課金できる設計）

## 4. ゆめのきの差別化軸

### 4-1. 技術的差別化（ポートフォリオ価値）

| 差別化軸 | 詳細 |
|---|---|
| **Stripe Webhook 本番稼働** | `checkout.session.completed` / `customer.subscription.deleted` / `invoice.payment_succeeded` の3イベントを本番で受信・永続化済み。冪等性・署名検証・KPIログまで実装 |
| **OpenAI 3サービス統合** | Chat（夢分析）・Audio（Whisper音声文字起こし）・Image（DALL-E画像生成）を同一アプリで統合 |
| **フルスタック構成** | Next.js 16 (App Router) + Rails 7.1 API + PostgreSQL を別ドメインで本番運用。CORS・JWT・HttpOnly Cookie のクロスドメイン設計が完成している |
| **フェアユース制限の設計** | プレミアム：AI分析1日50回・月次サマリー月3回・画像生成月31枚。`ai_usage_logs` テーブルで計測 |
| **キャラクター「モルペウス」** | 競合にはないオリジナルキャラクター。AIの口調・世界観を統一 |

### 4-2. ポートフォリオとしての差別化

競合は「使えるプロダクト」だが、**ゆめのきは「作れる根拠」**を示すポートフォリオでもある。

面接官に語れるポイント：
- 「なぜこう設計したか」を説明できる（Webhook冪等性・JWT設計・OpenAI統合）
- 本番で動いているものを見せられる（Vercel + Render）
- APIコストの変動費設計を意識した機能制限を実装した
- 競合調査を経て方針転換できる（プロダクト思考の証拠）

## 5. 今後の開発方針

### 方針A：競合から学んで取り込む（機能）

| 取り込み候補 | 優先度 | 理由 |
|---|---|---|
| UXの改善（フロー整理） | 🔴 高 | 競合のUIで気づいた導線の問題を修正 |
| 夢の検索・フィルター機能 | 🟡 中 | 記録が増えたときの使いやすさ |
| タグの可視化・分析 | 🟡 中 | 感情の変化を見える化（競合が強い部分） |

### 方針B：競合にない強みを伸ばす（差別化）

| 強化軸 | 具体的アクション |
|---|---|
| **ポートフォリオとしての説明力** | README を「面接官に見せる資料」として整備。設計の理由を言語化 |
| **音声入力の使いやすさ** | 競合にない機能として磨く（モバイルUX改善） |
| **モルペウスのキャラクター性** | 月次サマリーの文章・UIトーンで「この子がいるから使い続けたい」体験を作る |
| **透明な価格設計** | 無料枠・プレミアム枠の違いを明確にしてユーザーの信頼を得る |

### 方針C：アプリ名の刷新（実施予定）

- 表示名：`ユメログ` → **`ゆめのき`**
- 変更対象：`frontend/app/layout.tsx`・`manifest.json`・`README.md`
- 変更しない：バックエンドURL・DB・環境変数（本番稼働中のため）
- 理由：同名競合との混同を避け、独自のブランドを確立する

---

## 6. 競合発見から得た教訓

1. **アプリ開発前に競合調査を行う**：名前・機能・価格のリサーチを先にする
2. **公開されているアプリから積極的に学ぶ**：UI・UX・機能設計のベンチマークとして活用する
3. **「ショック = 情報」**：競合の存在は脅威ではなく、ユーザーニーズと市場の証拠
4. **ハデスの死にゲー原則**：スキルは消えない。再スタートは必ず前より速くなる
